Day 14/30 — Mini Project & Review

Building a Local Document Q&A App

Assembling chunking, vector embeddings, pre-filtering, grounding prompts, templates, and memory into a complete production-ready application.

1. Week 2 Review: The Architecture

In Week 2, we covered the core building blocks of production retrieval architectures. A robust Document Q&A application integrates all of these steps:

  1. Document Ingestion & Chunking (Day 9): Documents are loaded and split by natural headings with a 1,000-char window and 150-char overlap to preserve boundary context.
  2. Vector Space Representation (Day 8): Text segments are converted to mathematical coordinate vectors to calculate similarities.
  3. Metadata Filtering & Retrieval (Day 11): Unrelated document types (like log outputs) are filtered out before scoring, retrieving only the most relevant Top-K candidates.
  4. Grounded Prompts (Day 10): System instructions force the model to answer only from the excerpts and refuse queries with insufficient references.
  5. Prompt Registry (Day 12): System and User prompt templates are compiled using Zod schemas to ensure input variables are valid at runtime.
  6. Stateful Memory (Day 13): Durable user preferences (like concise vs detailed styles) are loaded from a secure preference store and injected into the prompt.

2. Connecting to the Document Q&A Branch

On the feat/document-qa branch of my-ai-tools, we implemented this architecture in a modular Python project (in `document_qa/`):

  • FastAPI Backend (api.py): Exposes endpoints to upload documents, rebuild the local FAISS index, execute similarity retrieval, and ask questions.
  • Streamlit Frontend (streamlit_app.py): Provides a beautiful dashboard to manage uploaded files, tune Top-K/similarity thresholds, and converse with the document corpus with dynamic sources previews.
