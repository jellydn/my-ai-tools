Day 9/30 — Chunking

Document Splitting, Overlap, and Chunk Size

Why bad chunking strategies silently destroy the quality of AI retrieval systems, and how to fix them.

1. The Retrieval Dilemma

Large Language Models have finite context windows. To search a large corpus of text or a codebase, we must split it into smaller pieces called chunks. These chunks are embedded into vectors and stored for retrieval. However, splitting text introduces a tradeoff:

  • Too Large: Preserves broad context, but dilutes specific search terms. The model gets "distracted" by irrelevant text in the same chunk.
  • Too Small: Pinpoints exact words, but loses the surrounding meaning. The model gets a single sentence without knowing the subject.

The goal is to produce chunks that are small enough to be semantically dense, but large enough to stand alone as a complete thought.

3. Semantic Chunking (AST & Headings)

Standard chunking slices text at arbitrary lengths (with minor adjustments for newlines). This often cuts code blocks, functions, or markdown sections in half.

In PR 309, we introduced a Semantic Chunker (lib/code-taste/chunker.ts) that solves this for codebases:

  1. TypeScript (AST): Uses web-tree-sitter to identify top-level nodes (class, function, interface). It keeps declarations intact and prepends supporting type interfaces if referenced in the function body.
  2. Markdown: Slices documents by heading blocks (e.g. # Heading 1) to ensure sections remain together, while avoiding splitting code fences (```).
