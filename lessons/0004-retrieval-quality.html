Day 11/30 — Retrieval Quality

Optimizing RAG Retrieval: Top-K & Pre-Filtering

How tuning your search scope and applying metadata filters eliminates noise and saves prompt token costs.

1. The Retrieval Quality Problem

A RAG application is only as good as the documents it retrieves. If the retrieval step fails to fetch the supporting facts, even the largest LLM will fail to answer correctly. However, optimizing retrieval is a balancing act:

  • Top-K Trade-off: Retrieving more chunks (higher K) increases the likelihood of finding the answer (answer-support rate), but floods the prompt with irrelevant noise and exponentially increases token billing.
  • Relevance Density: The percentage of retrieved chunks that are actually relevant to the question. High density prevents LLM distraction and lowers costs.

2. Connecting to PR 317

In PR 317 (feat: evaluate and improve RAG retrieval quality), we implemented evaluation benchmarks and performance enhancements:

  • Metadata Pre-filtering: Enabled filtering by document type (e.g. documentation, source, pull_request) before computing vector similarities. This eliminates irrelevant directories (like logs or tests) completely.
  • Evaluation Benchmark: Added a local reproducible benchmark in scripts/evaluate-retrieval.ts to measure metrics like relevance density and prompt token budgets across different Top-K values.

Key Finding: Moving from Top 3 to Top 5 improved answer support from 67% to 100%, but increasing K to 20 ballooned token consumption by 7.5× without yielding additional relevant details.
