Day 10/30 — RAG Basics

Retrieval-Augmented Generation & Grounding

How the Retrieve-Assemble-Generate pipeline keeps LLMs factually grounded in your local data.

1. The RAG Pipeline

A standard LLM only knows the data it was trained on. To answer questions about local, private, or real-time data, we use Retrieval-Augmented Generation (RAG). The process follows three distinct steps:

  1. Retrieve: The system embeds the user's question, performs vector search (Day 8), and fetches the most relevant text chunks (Day 9).
  2. Assemble: The system wraps the retrieved text excerpts along with a strict system instruction (the grounding policy) and the user's query into a structured context message.
  3. Generate: The model generates the answer, drawing details exclusively from the assembled context.

Important: Grounding prompts instruct the model to act as a strict reader rather than using its own parametric memory. If the information is not in the text, it must refuse to answer.

2. Connecting to PR 315

In PR 315 (feat: harden repository RAG knowledge base), we implemented and hardened this pipeline directly in the codebase:

  • Grounded Prompting: The server uses the RAG_SYSTEM_PROMPT defined in lib/rag-prompt.ts, instructing the model to answer only from the excerpts and return "This is not documented in the repository." on failure.
  • Zero-LangChain Transparency: The entire process—document loading, 1,000-char chunking, cosine retrieval (top 5), and streaming assembly—is written in clean, explicit TypeScript without third-party frameworks.
  • Observability: Structured JSON logs trace query-to-chunk paths, similarity scores, token counts, and latency, making it simple to evaluate and debug.
