Day 13/30 — Memory

Short-Term & Long-Term Memory for AI Apps

How separating ephemeral conversation state from durable user preferences makes AI interactions personal, safe, and controllable.

1. Two Kinds of Memory

Every useful AI application needs memory, but not all memory is equal. The critical design decision is what to remember and for how long:

  • Short-term memory holds the current conversation messages, tool results, and working variables. It lives inside the context window and vanishes when the session ends. Think of it as the AI's working memory.
  • Long-term memory persists preferences, facts, and decisions across sessions and projects. It survives restarts, context window resets, and even tool changes. Think of it as the AI's notebook.

Key insight: short-term memory is cheap but fragile; long-term memory is durable but dangerous. Every preference you persist must pass through an explicit consent gate — the user must confirm it before it's stored. See MemGPT for a tiered memory architecture reference.

2. Connecting to the User-Memory MCP

On the feat/user-preference-memory branch, we built a cross-project user-memory MCP server that implements safe long-term memory:

  • Explicit-only writes: The set() method requires a confirmed: true flag. Calling with false throws: "Preferences may only be stored after the user explicitly states or confirms them."
  • Secret rejection: Keys matching patterns like apiKey, token, or password and values matching JWT, Bearer, or PEM patterns are blocked before they ever reach disk.
  • Atomic file writes: Preferences are written to a temporary file, then atomically renamed to prevent corruption from concurrent writers or crashes.
  • Value-free audit trail: Every mutation (set, delete, reset) is logged to audit.jsonl with the key name and timestamp but never the value, preserving accountability without leaking data.
  • Full user control: Users can list, delete individual keys, or reset all preferences at any time.
